Lightning is the emerging settlement rail for machine-to-machine payments, but most nodes don't route reliably and most L402 services don't have SLAs. SatRank measures every indexed endpoint from its own LND, decomposes the L402 contract into 5 stages (challenge → invoice → payment → delivery → quality), publishes a Bayesian posterior per stage, and ships a weekly signed calibration history on Nostr so any agent can verify the oracle's accuracy across time. Federated by design: any operator can run a SatRank-compatible oracle and cross-attest. No account, no API key.

Install. One call. Paid response.

The SDK resolves your intent against the live registry, picks the top-ranked endpoint inside your budget, handles the L402 handshake, pays the invoice with your wallet, and returns the service response. Free fallback to the next candidate on failure.

agent.ts 
// 1. npm install @satrank/sdk
import { SatRank } from '@satrank/sdk';
import { LndWallet } from '@satrank/sdk/wallet';

const sr = new SatRank({
  wallet: new LndWallet({
    restEndpoint: 'https://127.0.0.1:8080',
    macaroonHex: process.env.LND_MACAROON_HEX!,
  }),
});

// 2. One call: discover, pay, fetch.
const result = await sr.fulfill({
  intent: { category: 'data' },
  budget_sats: 50,
});

if (result.success) {
  console.log(result.response_body);       // paid API payload
  console.log(result.endpoint_used!.url);  // which endpoint served it
  console.log(result.cost_sats);           // total sats spent (invoice + fees)
}

Under the hood

Intent → ranked candidates → paid fulfillment.

Two HTTP hops. The first is free and rate-limited; the second is paid per request via L402. Everything is documented in /api/openapi.json, interactively browsable at /api/docs.

2

Pay the chosen endpoint

Your agent (or the SDK) calls the selected endpoint_url directly. The provider replies HTTP 402 Payment Required with a BOLT11 invoice in the WWW-Authenticate header. Pay the invoice with any LN wallet, then retry the request with Authorization: L402 <mac>:<preimage>. This is the standard L402 protocol. SatRank does not proxy the request or custody funds.

l402.sh 
# 1. First call returns 402 with an invoice
curl -i https://grid.ptsolutions.io/v1/intelligence/demand-supply/ercot
# HTTP/2 402
# WWW-Authenticate: L402 macaroon="AGIAJE…", invoice="lnbc250n1p…"

# 2. Pay the invoice with your LN wallet (returns a preimage)
lncli payinvoice lnbc250n1p…

# 3. Retry with the paid token
curl -H 'Authorization: L402 AGIAJE…:' \
     https://grid.ptsolutions.io/v1/intelligence/demand-supply/ercot
Trust model

Three layers: a free posterior, a 5-stage L402 contract decomposition, and a paid LN-graph breakdown.

Every candidate in /api/intent ships with the free Bayesian posterior plus the per-stage L402 contract decomposition (challenge / invoice / payment / delivery / quality). The 5-stage view tells your agent which step is likely to fail, not just whether the request will succeed end-to-end. The paid LN-graph component breakdown (1 sat) gives the deterministic node-level scoring for diligence callers.

4.1

Free · Bayesian posterior + 5-stage L402 contract

Each candidate carries the legacy bayesian.p_success posterior plus a stage_posteriors block decomposing the L402 contract into 5 independent Beta posteriors. The composed end-to-end probability is the chain rule product over stages with sufficient observations. http_method is persisted from the upstream registry so the SDK calls the right method without a 405-fallback round trip.

4.2

Paid · LN-graph scoring (5 components)

Distinct from the 5-stage L402 contract above: this is the deterministic node-level scoring used to seed the Bayesian prior. It answers "is this Lightning node well-connected, well-routed, well-aged?" — orthogonal to "does this paid endpoint deliver?". Five components, frozen public weights. The per-node numeric contribution is available via GET /api/profile/:id (1 sat per call).

25% Volume

50/50 blend of channel count (log-scaled, ref 500) and capacity in BTC (log-scaled, ref 50 BTC).

30% Reputation

Five sub-signals, weights sum to 100:

  • Sovereign PageRank centrality20%
  • Peer trust (BTC / channel)30%
  • Routing quality (hops + latency from our probes)20%
  • Capacity trend (7d delta)15%
  • Fee stability (changes per week)15%
15% Seniority

Days since first seen on the network. Exponential growth curve, half-life 730 days.

15% Regularity

Uptime (70%) + latency consistency (20%) + hop stability (10%).

15% Diversity

Unique peers (log-scaled, ref 500) or unique counterparties, whichever the data supports.

Post-composite modifiers: verified-tx bonus ×1.0 to 1.10, probe-failure penalty ×0.65 to 0.90 (graded by cause: dead / zombie / liquidity). Full constants in src/config/scoring.ts. Values are frozen; changes are announced before deployment.

Neutrality by construction

Three reasons SatRank can't play favourites.

01

AGPL-3.0, end to end.

The scoring engine, the crawler, the API: all open source under AGPL. If we ever changed the ranking to favour a partner, you would see it in the diff. You can also run your own SatRank and compare results node by node.

02

Deterministic scoring.

No ML, no opaque weights, no vibes. Five components, frozen public weights, inputs measured from our own LND node. Given the same probe data, two runs produce the same ranking, byte-for-byte.

03

Rate-locked pricing.

Deposit tiers are public, and the rate is burnt into the macaroon at deposit time. The business model cannot bias the ranking because the ranking is not what is sold, and it cannot reward individual callers because the tier is a function of the deposited amount, not of the caller's identity.

Pricing

Free directory. Paid freshness, paid batches.

Discovery is free. POST /api/intent, GET /api/agents/top, and the per-agent reads (/api/agent/:hash, /verdict, /history, /attestations) are rate-limited to 10/min/IP and require no payment. Free intent responses ship with an explicit advisory.freshness_status per candidate (fresh / recent / stale / very_stale) so your agent can act on staleness without trusting an opaque score.

Pay only when you need a guarantee or amortise. POST /api/intent?fresh=true costs 2 sats and forces a synchronous HTTP probe on the top candidates so last_probe_age_sec is < 60 s on every returned row. Batch verdicts, full profiles, and end-to-end L402 probes stay paid because they amortise round-trips or use SatRank's own LND. Per-call invoices are 1 sat by default; deposit one of five public tiers to burn a discounted rate into your macaroon.

Nostr-native distribution

Every posterior, every calibration, every outcome — signed, on public relays.

SatRank publishes its trust graph + calibration history + transferable assertions to relay.damus.io, nos.lol, and relay.primal.net. Any Nostr client can subscribe, verify the Schnorr signatures, and stream updates in real time. The same Nostr surface lets other oracles federate (kind 30784 announcement) and any agent publish outcomes (kind 7402 with PoW + preimage proof).

Federation

Run your own oracle. Cross-attest the others.

SatRank is one of N+1 SatRank-compatible oracles. The federation discovery protocol (kind 30784) and the calibration cross-attestation pipeline (kind 30783) let any operator spin up an independent oracle, publish its own posteriors, and have agents aggregate the network with weighted Bayesian model averaging — no single oracle can capture the trust signal. Operator quickstart →

01

Discovery built-in.

Subscribe to kind 30784 on the canonical relays and you get every SatRank-compatible oracle within 24 hours of announcement. GET /api/oracle/peers exposes the local snapshot. The SDK ships aggregateOracles() with built-in trust filters (max staleness, minimum catalogue size, calibration history required).

02

Cross-attested calibration.

Each oracle publishes a kind 30783 weekly calibration event: delta_mean, delta_p95, n_endpoints. SatRank ingests every peer's calibration and exposes the full history at GET /api/oracle/peers/:pubkey/calibrations. An oracle that drifts is visible to the entire federation.

03

Web of trust via kind 7402.

Any agent who has paid for an L402 endpoint can publish a kind 7402 outcome report (with PoW + preimage proof). Every oracle ingests the report with Sybil-resistant weighting (base × pow_factor × identity_age × preimage_factor), and after a 1h anti-spam delay the report is consolidated into the per-stage posteriors. The trust signal grows with the agent network, not just with the oracle's own probes.

Thesis

Why this, why now.

01

Agents will need to pay strangers.

As autonomous agents start calling third-party APIs, they will pay for most of them. The cheapest, most auditable settlement rail for machine-to-machine micropayments is Lightning, and the emerging protocol is L402.

02

Every payment is a trust decision.

Most LN nodes advertised in the gossip graph never route a payment. SatRank probes each one from its own LND. Before an agent spends a sat, it needs to know: will this endpoint actually respond? Is the price fair? Has it failed its last 50 callers? A model can't guess. It needs a measured, cryptographically-signed answer.

03

The oracle has to be sovereign + federated.

An AI-native trust oracle can't live behind a trusted API gateway. It is the gateway. SatRank runs its own bitcoind, its own LND, its own Nostr identity. Beyond sovereignty, the oracle has to be federated: any operator can run a SatRank-compatible instance, cross-attest calibration, and an agent aggregates N oracles weighted by their published accuracy. If any single oracle goes dark, the federation continues — and the last weeks of signed calibration history remain verifiable on public relays.
