1. Overview

SatRank is a trust oracle for the Bitcoin Lightning Network. It measures every reachable node, crawls every L402 service endpoint it can discover, and combines three independent observation streams into a Bayesian posterior over the event "a request to this target will succeed." Agents query the oracle with a typed intent (category, budget, latency ceiling), and receive a ranked shortlist of endpoints with the posterior, a confidence interval, recent-activity counts, and a risk profile.

The oracle is run from a single full Bitcoin node with its own LND, which is the primary source of truth: the LND graph feeds node discovery, queryRoutes probes feed reachability, and the same daemon holds the macaroon that decodes BOLT11 invoices exposed by L402 services. No third-party API sits on the critical path of a score.

What this document is

  • A reference for the exact math used to compute p_success, ci95_low, ci95_high, and the scalar score.
  • A catalogue of every public HTTP endpoint, its authentication model, and its rate limits.
  • The Nostr event schema SatRank publishes, and how third parties can reconstruct verdicts from those events alone.

What this document is not

  • A product pitch. Marketing copy lives on the landing.
  • A guarantee. All formulas and constants below may change; when they do, the schemaVersion in /api/health increments and the change is annotated in the repository.
  • Financial advice. The oracle emits probabilities, not recommendations.

2. Measurement surface

The oracle maintains four relational populations, all keyed on sha256(public_key) (the canonical public_key_hash, 64 hex chars) or on a URL hash (sha256(canonical_url)).

2.1 Lightning nodes

Every node observed in the LND gossip graph is indexed. The crawler ingests GET /v1/graph from the oracle's own LND and upserts agents rows with alias, color, public key, first-seen and last-seen timestamps, channel count, aggregate capacity, and disabled-channels count. PageRank is recomputed hourly over the full graph. Live counts (captured at document build time from /api/stats): 8,201 agents indexed, 74,689 channels, 9,490.2 BTC aggregate public capacity.

2.2 L402 service endpoints

The registry crawler walks https://402index.io/api/v1/services?protocol=L402, plus any endpoint submitted through POST /api/services/register. For each URL, it issues a safe HTTPS GET (SSRF-hardened via fetchSafeExternal), expects a HTTP 402 Payment Required response with a WWW-Authenticate header, extracts the embedded BOLT11 invoice, decodes it through the local LND's DecodePayReq, and writes the mapping (url_hash, operator_pubkey, price_sats, source). Live count: 208 endpoints from 402index, across 27 categories (largest: data 49, energy/intelligence 48, video 35).

2.3 Probes

Reachability is measured by issuing QueryRoutes from the oracle's LND at four liquidity tiers: 1 000, 10 000, 100 000, 1 000 000 sats. A probe is reachable when LND returns at least one route; unreachable otherwise. Each probe persists hops, latency, amount, reachable flag, and timestamp. A rolling ring of 24 h activity holds 895,786 probes at the time of writing.

2.4 Reports

A client that has already paid an L402 invoice can submit the preimage back to SatRank as a first-party report. The preimage proves payment, and the oracle's token_query_log scopes the report strictly to the target that the client's token was issued for. Tiered weighting (Section 3) distinguishes anonymous preimage-only reports from NIP-98 signed reports.

2.5 Attestations

Nodes can attest to one another with a signed score in [0, 100]. The oracle stores attester, subject, category (including successful_transaction, failed_transaction, unresponsive), score, and timestamp. Anti-gaming machinery (cycle detection at depth 3 and depth 4, age gating, mutual attestation penalty) is applied at scoring time (see Section 5).

2.6 Database schema

Schema version 41. Twelve data tables: agents, transactions, attestations, score_snapshots, probe_results, channel_snapshots, fee_snapshots, token_balance, service_endpoints, service_probes, token_query_log, report_bonus_ledger, plus streaming posteriors tables (endpoint_streaming_posteriors, node_streaming_posteriors) and endpoint_daily_buckets for recent-activity counts. Engine: PostgreSQL 16 (migrated from SQLite in Phase 12B).

3. Bayesian trust posterior

For each target T (either a node public_key_hash or an endpoint url_hash) the oracle maintains a posterior on the Bernoulli success probability of a future request. The Beta-Binomial core is conjugate: the prior is Beta, the likelihood is Binomial, so the posterior is Beta with analytic updates.

3.1 Prior

The base flat prior is Beta(α₀ = 1.5, β₀ = 1.5), defined in src/config/bayesianConfig.ts. The choice is weakly informative: the mode lies in the interior of [0, 1] (not at 0.5 nor at an endpoint), so a handful of observations move the posterior decisively but a single observation does not. When hierarchical information is available (Section 4), the base prior is replaced by the hierarchical prior.

3.2 Observation sources and weights

Three streams update the posterior, each with a source-level weight reflecting the evidential strength of the observation:

SourceWeightSemantics
probe1.00Sovereign LND probe (queryRoutes at one of four tiers).
paid2.00First-party success proven by a valid preimage.
report, tier LOW0.30Anonymous report without preimage.
report, tier MEDIUM0.50Preimage-backed anonymous report.
report, tier HIGH0.70Preimage plus scoped token_query_log token.
report, tier NIP981.00NIP-98 HTTP-auth signed by a known Nostr pubkey.

3.3 Exponential decay

Each observation at time t_obs is reweighted at read time with the exponential decay factor 

w(Δt) = exp(-Δt / τ)    where Δt = t_now - t_obs and τ = 7 days

Reads are served against streaming_posteriors, a per-source table that caches the running sums Σ wᵢ · sᵢ (successes) and Σ wᵢ · fᵢ (failures), decayed to the query timestamp. No full replay of the transactions log is required. Buckets older than BUCKET_RETENTION_DAYS = 30 are purged.

3.4 Posterior

Let sᵢ ∈ {0, 1} be the outcome of observation i and wᵢ the product of source weight and decay. For each source k, the oracle computes 

α_k = α₀ + Σᵢ∈k wᵢ · sᵢ
β_k = β₀ + Σᵢ∈k wᵢ · (1 - sᵢ)

The combined posterior adds the excess evidence mass from each source on top of a single shared prior, which avoids double-counting when multiple sources have data: 

α = α₀ + Σ_k max(0, α_k - α₀)
β = β₀ + Σ_k max(0, β_k - β₀)

p_success = α / (α + β)
n_obs     = Σ_k nObsEffective_k    (each source contributes its own Σ wᵢ)
ci95_low  = BetaInvCDF(0.025; α, β)
ci95_high = BetaInvCDF(0.975; α, β)

BetaInvCDF is the inverse regularized incomplete Beta function; the implementation lives in src/utils/betaBinomial.ts and is covered by a property-based test suite against reference values.

3.5 Verdict

The scalar verdict collapses the posterior into one of four discrete states. Evaluation is strictly ordered, so the first matching rule wins:

VerdictCondition
INSUFFICIENTn_obs < 1 (no evidence from any source)
RISKYp_success < 0.50 or ci95_high < 0.65
UNKNOWNci95_high - ci95_low > 0.40 or n_obs < 10
SAFEp_success ≥ 0.80 and ci95_low ≥ 0.65 and n_obs ≥ 10 and converged
fallbackAnything else returns UNKNOWN.

Converged means at least CONVERGENCE_MIN_SOURCES = 2 independent streams report p_success ≥ CONVERGENCE_P_THRESHOLD = 0.80. A single loud source cannot push a verdict to SAFE on its own. The verdict reason is returned alongside the verdict in verdict_reason and mirrors the matched rule.

3.6 Verdict overlay

The API-level verdict composition in verdictService.ts additionally escalates bayes.verdict to at least RISKY when evidence-based flags fraud_reported or negative_reputation are present. Evidence of harm dominates posterior uncertainty.

4. Prior cascade & risk profile

4.1 Hierarchical prior

A fresh target has no direct evidence, so the posterior would collapse to the flat prior and every verdict would be INSUFFICIENT. To share information across related targets, the oracle resolves the prior through a four-level cascade implemented in bayesianScoringService.resolveHierarchicalPrior:

  1. operator. Sum the decayed streaming posteriors of every endpoint the target's operator has claimed and verified. The operator layer's mass is scaled by OPERATOR_PRIOR_WEIGHT = 0.5 before being used as the prior, because operator-level evidence is one level above the target.
  2. service. Same as above, restricted to endpoints that share the target's service hash.
  3. category. Same, restricted to endpoints in the same category (as reported by 402index). Sibling hashes are discovered at query time via serviceEndpointRepo.listUrlHashesByCategory.
  4. flat. Beta(1.5, 1.5).

The cascade is short-circuited at the first level that exceeds PRIOR_MIN_EFFECTIVE_OBS = 30 effective observations. Below that threshold the next level is tried. If no level qualifies, the flat prior is used. The resolved level is returned in the API response as prior_source, and a trace of the cascade is available via /api/agent/:hash/verdict?debug=prior.

4.2 Risk profile

The Bayesian posterior is a snapshot; it does not indicate whether a target is degrading. The risk profile is a separate Option-B trend signal computed by comparing two non-overlapping windows on the daily_buckets table: 

recent  = success_rate over the last 7 days
prior   = success_rate over the preceding 23 days  (total window = 30 days)
delta   = recent - prior

risk_profile =
  "high"    if delta ≤ -0.25
  "medium"  if delta ≤ -0.10
  "low"     if delta > -0.10
  "unknown" if n_obs(recent) < 5  (MIN_N_OBS)

A node that was stable at p_success = 0.95 last month and dropped to 0.60 this week is risk_profile = "high" even though its long-run posterior still rounds up. An agent fetching a verdict for a time-sensitive workflow should read both the posterior and the risk profile.

4.3 Recent activity

For context, the API also returns a compact rolling count from endpoint_daily_buckets: 

recent_activity = {
  last_24h: Σ n_obs over the last 24 hours,
  last_7d:  Σ n_obs over the last 7 days,
  last_30d: Σ n_obs over the last 30 days
}

These are raw observation counts summed across the three sources, with no decay, intended for UI badges ("54 observations in the last 24 h") rather than for scoring.

5. Deterministic scoring v18

The Bayesian posterior is the oracle's public signal. Alongside it, a deterministic composite score S in [0, 100] feeds the internal hierarchical prior, the per-attester credibility weights used in anti-gaming, and an audit view exposed as components in /api/agent/:hash. It is not a success probability. It is a node-quality signal built from structural facts about the Lightning graph and the oracle's own probe history.

The scoring implementation is in src/services/scoringService.ts (817 lines). The constants below are in src/config/scoring.ts; the current version is v18, frozen on 2026-04-22 in commit c02c264 (Phase 12C).

5.1 Composite

S = round(
    w_V · Volume       +
    w_R · Reputation   +
    w_S · Seniority    +
    w_Reg · Regularity +
    w_D · Diversity
)

w_V   = 0.25   WEIGHT_VOLUME
w_R   = 0.30   WEIGHT_REPUTATION
w_S   = 0.15   WEIGHT_SENIORITY
w_Reg = 0.15   WEIGHT_REGULARITY
w_D   = 0.15   WEIGHT_DIVERSITY
Σ     = 1.00

5.2 Volume (25 %)

A 50 / 50 blend of channel count and aggregate capacity. 

channelScore  = min(100, log(channels + 1) / log(501) · 100)
capacityScore = btc > 0 ? min(100, log(btc + 1) / log(51) · 100) : 0
volume        = round(0.5 · channelScore + 0.5 · capacityScore)

Reference points: 500 channels → 100; 50 BTC of capacity → 100. The blend defuses the "thousands of 1-sat channels" attack that scored 100 under pre-v15 formulations.

5.3 Reputation (30 %)

For Lightning nodes, Reputation is a weighted sum of five sub-signals, with dynamic renormalization when centrality or peer-trust data is structurally unavailable (fresh nodes, pre-crawler backfill).

Sub-signalNominalFormula
Centrality0.20Sovereign PageRank over the full LND graph. Fallback: LN+ hubness / betweenness ranks.
Peer trust0.30min(100, log₁₀(btcPerChannel · 100 + 1) / log₁₀(201) · 100)
Routing quality0.200.6 · hopScore + 0.4 · latencyScore from our probes (hopScore = max(4, 100 - (meanHops - 1) · 12), latencyScore = max(0, 100 - meanLatencyMs / 3))
Capacity trend0.15100 / (1 + exp(-6 · delta7d)) over channel_snapshots; neutral 50 when no history.
Fee stability0.15100 / (1 + exp(1.5 · (changes_per_channel_per_week - 2))); neutral 50 when no fee data.

When a sub-signal is structurally unavailable (versus measured at 0), its weight is dropped and the remaining weights are renormalized to sum to 1. This is the availSum mechanism in computeLightningReputationBreakdown: only Centrality and Peer trust can be unavailable; the other three always carry a semantically meaningful neutral fallback of 50, so they stay in the average.

For agent-to-agent Reputation (based on attestations, not graph data), the same function returns a weighted average of attestation scores with exponential decay (ATTESTATION_HALF_LIFE = 30 days), multiplied by the attester's own avg_score / 100, plus a report signal in [-REPORT_SIGNAL_CAP, +REPORT_SIGNAL_CAP] = [-10, +10]. The cycle-detection machinery (mutual, 3-hop, 4-hop) is applied before aggregation.

5.4 Seniority (15 %)

seniority = round(100 · (1 - exp(-days_since_first_seen / 730)))

Half-life is two years. At day 1, ≈ 0.14 points. At one year, ≈ 37. At two years, ≈ 63. Plateau at 100 around day 3 000.

5.5 Regularity (15 %)

For nodes with ≥ 3 probes in the last 7 days: 

uptime              = reachable_probes / total_probes  over 7 days
latency_consistency = exp(- stddev_latency / mean_latency)   (1 = rock steady)
hop_stability       = 1 - min(1, stddev_hops / 3)

regularity = round(uptime · 70 + latency_consistency · 20 + hop_stability · 10)

Fallback for fresh nodes (< 3 probes): gossip recency with a 90-day half-life. The fallback saturates fast and is explicitly earmarked as provisional in the components breakdown.

5.6 Diversity (15 %)

diversity = uniquePeers > 0
          ? min(100, log(uniquePeers + 1) / log(501) · 100)   // primary
          : capacityFallback(btc)                              // only if peer count unknown

Reference point: 500 distinct peers → 100. A hub with 100 BTC split across two counterparties scores poorly on diversity because it contributes little routing optionality.

5.7 Post-composite modifiers

Two multiplicative modifiers are applied after the weighted sum, on the real-valued totalFine track (kept at 2-decimal precision alongside the integer score): 

// Probe penalty (anti-gossip-spam)
if base tier (1k sats) is unreachable:
    probeMult = 0.65   // disabledRatio ≥ 0.80 (dead)
              | 0.70   // gossip age > 30 days (zombie)
              | 0.80   // gossip age > 7 days OR disabledRatio ≥ 0.30
              | 0.90   // otherwise (liquidity-only)
else:
    probeMult = weightedAvg(success_rate at tiers 1k, 10k, 100k)
                with weights {1k: 0.4, 10k: 0.3, 100k: 0.2}
                floor = 0.65, no-op if ≥ 1.0

// Verified-tx bonus
bonus = 1 + min(VERIFIED_TX_BONUS_CAP, verifiedForBonus) · 0.003
      ∈ [1.0, 1.10]

total = round(max(0, totalFloat · probeMult · bonus))

The 1 000 000-sat probe tier is measured and exposed via maxRoutableAmount, but is excluded from the penalty weighting: a failure at 1 M sats often reflects a legitimate liquidity limit rather than a defect.

5.8 Confidence levels

confidence ∈ {very_low, low, medium, high, very_high}, derived from total_transactions + total_attestations_received with breakpoints at 5 / 20 / 100 / 500. The composite score is only intended to be compared across agents with the same confidence tier.

6. Verdict & intent API

6.1 Intent resolution

An agent describes what it needs and the oracle returns a ranked shortlist. The request shape: 

POST /api/intent    (free, no authentication, no quota)
Content-Type: application/json

{
  "category":       "energy/intelligence",
  "keywords":       ["ercot", "grid"],
  "budget_sats":    1000,
  "max_latency_ms": 3000,
  "limit":          5
}

Valid categories are enumerated at GET /api/intent/categories. The resolver filters service_endpoints by category, then by budget (service_price_sats ≤ budget_sats), then by 7-day median HTTP latency, then enriches each candidate with a Bayesian block, flags, reachability, and advisory, and sorts by (p_success DESC, ci95_low DESC, price_sats ASC).

6.3 Per-target verdict

GET /api/agent/:public_key_hash/verdict returns the full verdict for a single target, including the bayesian posterior, sources breakdown, convergence, advisory, personal trust (when a caller pubkey is provided), and pathfinding (when the caller can be located in the Lightning graph). This endpoint is L402-gated (apertureGateAuth then balanceAuth): anonymous callers see a 402 with a BOLT11 for 1 sat.

7.2 L402 flow

Paid endpoints are guarded by a thin Express middleware layered in front of Aperture. The first request returns: 

HTTP/1.1 402 Payment Required
WWW-Authenticate: L402 macaroon="MDAyN…", invoice="lnbc10n1p…"

The client pays the BOLT11 invoice with any Lightning wallet, captures the preimage, and retries the request with: 

Authorization: L402 <macaroon>:<preimage>

The middleware validates that sha256(preimage) == payment_hash embedded in the macaroon, marks the token as spent, and forwards.

7.3 Deposit (variable-amount token, tiered rate)

POST /api/deposit issues a BOLT11 invoice of N sats in the range [21, 1 000 000]. The client pays and posts the preimage back; the oracle looks up the deposit tier whose floor is the highest floor less than or equal to N and credits N / rate requests to a token identified by that preimage, where rate is the tier rate in sats per request. Tier rates are public at GET /api/deposit/tiers and currently run 1.0, 0.5, 0.2, 0.1, 0.05 sat per request across the five tiers. The rate is engraved on the token at creation: a later edit to the deposit_tiers table does not affect existing tokens. The token format is Authorization: L402 deposit:<preimage>. Rate limit: 3 invoices per minute per IP. Nginx routes L402 deposit:* directly to Express, bypassing Aperture, because the invoice is generated by our own LND through the invoice-only macaroon.

7.4 Preimage-as-report

A client that has already paid any L402 invoice (ours or a third-party endpoint discovered through /api/intent) can reuse that preimage as a report. POST /api/report accepts the preimage as authentication; the oracle matches it against token_query_log to confirm the report's scope is the same target the client was last directed to. A report signed with NIP-98 (Nostr HTTP auth) carries the highest source weight (1.00) because the signer's Nostr pubkey is reusable in other evidence.

7.5 Cross-cutting invariants

  • SSRF-hardening. Every outbound HTTP fetch from the crawler goes through fetchSafeExternal, which performs connect-time DNS validation and rejects private IPs before opening the socket.
  • Rate limiting. Per-token, per-IP, and global limits are enforced at the Express layer (writeRateLimit, reportRateLimit, depositRateLimit, probeRateLimit), and Aperture enforces L402 budget at the edge.
  • Circuit breakers. All calls to LND go through src/utils/circuitBreaker.ts. A trip opens for 30 s after 5 consecutive failures, returning a fast 503 instead of letting the event loop stack.
  • Semaphore. queryRoutes is globally capped at 10 in-flight and 50 queued requests. Excess requests reject with SemaphoreFullError, never OOM.
  • Deterministic ordering. All API responses use a fixed sort order; clients never need to re-sort server output.

8. Nostr publication

SatRank publishes verdicts over Nostr so third parties can reconstruct the trust posterior without querying SatRank's HTTP API. Publication is addressable (NIP-33): each target has exactly one live event per kind, updated in place, and any relay that has the latest event holds the current state.

Relays (DEFAULT_NOSTR_RELAYS, src/nostr/relays.ts):

8.4 Republication rules

A cycle scans streaming_posteriors every 15 minutes for entities whose last_update_ts falls inside the scan window. For each entity, shouldRepublish compares the current snapshot against nostr_published_events and emits a new event when any of the following is true:

  • First publish. No prior event for this entity.
  • Verdict transition. Current verdict differs from the last published verdict.
  • Advisory transition. Current advisory_level differs.
  • p_success change. |Δp_success| ≥ 0.005.
  • Heartbeat. Last publish is older than one day.

Even when shouldRepublish returns true, a fast-path hash check rebuilds the event template and compares payload_hash to the cached one. If they are identical (tag rounding collapsed the delta), no network call is made.

8.5 Verdict flashes (kind 20900)

On any verdict transition, a best-effort ephemeral event is also broadcast. Flashes are not persisted by relays and carry from_verdict and to_verdict tags in addition to the Bayesian block. Subscribers can react to trust changes without polling the addressable events.

8.6 NIP-13 proof of work

Certain relays (observed: nos.lol on some kinds, with a 28-bit target) require NIP-13 proof of work. The publisher mines the required number of leading zero bits on id using worker_threads, appends a nonce tag, and publishes. Kinds whose PoW target exceeds a configured ceiling are skipped on that relay rather than delaying the whole cycle.

8.7 Reconstructing state from Nostr alone

A subscriber that wants to re-derive a verdict from relays fetches all kind 30382 / 30383 / 30384 events authored by the SatRank npub, keeps the latest created_at per d tag, and reads the Bayesian block. The result is identical to /api/agent/:hash/verdict up to the time difference between the last publish and the HTTP query.

9. Operator hierarchy

An operator is an external entity (an individual, a company, a DAO) that claims ownership of one or more Lightning nodes or L402 endpoints. The operator layer is the coarsest level of the prior cascade (Section 4), and it also controls who can emit NIP-85-scoped reports about their own resources.

9.1 Lifecycle

  1. Register. The operator issues a NIP-98 HTTP-auth-signed POST /api/operator/register with { npub, display_name, resources: [{ type: "node" | "url", id }] }. The oracle creates an operator_record with status = "pending".
  2. Verify. For each claimed resource, the operator proves control out-of-band: a node claim is verified by signing a payload with the node's LN key (lnnode/signMessage); an endpoint claim is verified by serving a signed well-known file at the URL's origin. On success, the resource row moves to status = "verified".
  3. Endorse. Once verified, all endpoints under that operator inherit an operator-level prior (weighted 0.5). The operator can emit NIP-85 reports scoped to their own resources with the maximal source weight (1.00).

9.2 Resource lookup

Given a node public key or a URL hash, operatorService.resolveOperatorForNode returns { operatorId, status, displayName }. The lookup populates the operator_id field in /api/intent and /api/agent/:hash/verdict responses only when status === "verified". Pending or rejected operators trigger an advisory (OPERATOR_UNVERIFIED) rather than exposing the claim.

9.3 Prior effect

When an operator is verified on at least two resources with enough evidence to cross PRIOR_MIN_EFFECTIVE_OBS = 30, a fresh resource claimed by the same operator uses the operator's cumulative posterior as its prior (scaled by OPERATOR_PRIOR_WEIGHT = 0.5). A brand-new endpoint from a well-behaved operator therefore starts above the flat prior, but not at the operator's current p_success. A brand-new endpoint from an operator whose other resources are RISKY starts below the flat prior.

9.4 Anti-abuse

  • Claim shadowing. Self-submitted endpoint metadata (POST /api/services/register) can only fill empty fields. The trusted 402index crawler is the source of truth for name, category, and description; a self-submitter cannot overwrite them.
  • Operator takeover. A node or URL that already has a verified operator cannot be claimed by another operator without a signed revocation from the incumbent.
  • Decay on silence. If an operator's resources stop being probed successfully for 30 consecutive days, the operator-level prior decays back towards the flat prior on the same τ = 7 d schedule as the target-level posterior.
Why an agent can trust the oracle

10. Neutrality is a property of the mechanism, not a policy.

10.1 Deterministic scoring function

For any given input X (probes, reports, routes), there is exactly one output (α, β). The function f: (probes, reports, routes) → (α, β) has no randomness, no manual override, no admin panel. It is implemented in src/services/scoringService.ts and src/services/bayesian*.ts. Its weights are frozen in src/config/scoring.ts and changes are announced in CHANGELOG.md.

10.2 AGPL-3.0 as verification mechanism

Anyone can fork the repo, run SatRank locally against the same gossip data, and compare outputs node by node. Any deviation indicates either a bug or tampering. Mathematical reproducibility becomes social verifiability. The AGPL obligation to publish source for any modified deployment closes the loophole where a private fork could be used to replace the oracle covertly.

10.3 Mechanical volume-based pricing

Pricing is tiered, but every tier boundary is public and every rate is deterministic. The five tiers (21, 1000, 10000, 100000, 1000000 sats) are stored in deposit_tiers, a read-only seeded table published in the schema. When an agent deposits N sats, the handler looks up the tier whose floor is the highest floor less than or equal to N and writes that tier's rate into the deposit record linked to the payment hash. The rate cannot be changed after the fact: subsequent deposit_tiers edits would not affect existing deposits. The business model cannot bias the ranking because the ranking is not what is sold, and it cannot reward individual callers because the tier is a function of the deposited amount, not of the caller's identity. Two agents depositing the same amount receive the same rate, always.

10.4 What operators can and cannot do

  • Can: improve reliability, add capacity, run their own probes to self-monitor, submit self-registration metadata for their L402 endpoints.
  • Cannot: pay for rank, see their own score before it is computed and published, request manual correction, negotiate weight, appeal a posterior value.

10.5 Reproducibility and audit

Scoring version v18, frozen 2026-04-22 (commit c02c264, Phase 12C). Schema version v41. Legacy v2 endpoints retired. Audit surfaces exposed to agents:

  • The components field in GET /api/agent/:hash (paid, 1 sat) exposes the five-component decomposition with dynamically renormalized weights.
  • The sources field exposes the breakdown of evidence behind the posterior.
  • The Nostr stream exposes the full state continuously on public relays.
  • The token_query_log records every paid call with timestamp and target, queryable by the caller via their L402 macaroon.

10.6 Anti-gaming defenses

  • Sybil resistance: seniority + verified-tx modifiers dampen newly created node farms.
  • Volume manipulation: graph-topology detection flags self-transactions.
  • Probe manipulation: agents cannot influence probes, since probes originate from SatRank's own LND node.
  • Cycle detection (mutual, 3-hop, 4-hop) filters out reputation loops in attestation aggregation.

Where SatRank is honest about its weaknesses

11. What the oracle does not guarantee.

11.1 Empirical, not theoretical

Scoring is calibrated against observed data. There is no formal game-theoretic proof that the mechanism is attack-proof under all adversary models. Known attack surfaces are documented and mitigated (see section 10.6), but novel attacks are always possible and the response is to patch publicly, not to hide.

11.2 Coverage gaps

A significant fraction of Lightning nodes advertised in the public gossip graph never route a payment. The live share is exposed as phantomRate via /api/stats. SatRank does not score unreachable nodes: a truly unreachable node does not appear in the posterior database at all, which is the correct behavior but means agents should not treat absence from the database as a signal of badness, only of silence.

11.3 Provider-side dependencies

SatRank depends on external services. The 402index registry for L402 endpoint discovery, the LND gossip network for node discovery, and the providers themselves for BOLT11 invoice generation. If these degrade, SatRank degrades. Observed example: a provider returning HTTP 500 for its L402 challenge will show n_obs = 0 on its endpoints until recovery (the posterior has no signal to update). See docs/phase-13d/PHASE-13D-REPORT-2026-04-22.md for detailed case analysis.

11.4 Historical, not predictive

The posterior reflects historical behavior. A node that was reliable yesterday may fail tomorrow. The ci95 interval is a measure of historical uncertainty given the observed data; it is not a guarantee about future performance. Agents making high-value decisions should treat p_success as a probability density, not a commitment.

11.5 LND string-matching fragility

The circuit breaker carve-out (described in section 7) depends on matching specific error string patterns from LND v0.20.1 ("invalid index", "checksum failed", "failed converting data", "invalid character not part of charset"). An LND version upgrade may rephrase these errors, causing the breaker to re-trip on malformed provider invoices. This fragility is flagged for re-validation on any LND version bump.

11.6 Single-position probing

All reachability probes originate from SatRank's own LND node (pubkey 024b550337...). An agent operating from a different graph position may experience different success rates. The posterior reflects SatRank's viewpoint. For agents requiring position-personalized probes, the roadmap includes a walletProvider parameter that re-runs queryRoutes from the agent's declared position; check /api/openapi.json for current availability.

11.7 Anti-claim

What SatRank does NOT claim to do. It is not a KYC system. It is not a legal attestation of the operator's identity or good standing. It is not a guarantee of service quality, only a probabilistic estimate given observed behavior. An agent making high-value or legally sensitive decisions should use SatRank as one signal among several, not the only signal. No warranty, express or implied. See LICENSE (AGPL-3.0) section 15.
